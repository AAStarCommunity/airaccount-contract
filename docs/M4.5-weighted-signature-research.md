# M4.5 — Weight-Based Multi-Signature Research Note (F45)

## Problem

Current cumulative signature model uses fixed tiers:
- Tier 1: P256 only
- Tier 2: P256 + BLS
- Tier 3: P256 + BLS + Guardian

This is rigid — what if a community wants different security profiles?

## Proposed Weight Model

Each signature source has a configurable weight:

| Source | Default Weight | Notes |
|--------|---------------|-------|
| Owner passkey (P256) | 3 | Device-bound, always required as base |
| Owner ECDSA (EOA/KMS) | 2 | TEE-protected, auto-signs with passkey |
| DVT BLS aggregate | 2 | N-of-M distributed nodes |
| Guardian 1 | 1 | Family/trusted person |
| Guardian 2 | 1 | Family/trusted person |
| Guardian 3 (Community) | 1 | Community Safe multisig |

### Threshold Configuration

```solidity
struct WeightConfig {
    uint8 passkeyWeight;     // Default 3
    uint8 ecdsaWeight;       // Default 2
    uint8 blsWeight;         // Default 2
    uint8[3] guardianWeight; // Default [1, 1, 1]
    uint8 tier1Threshold;    // e.g., 3 (passkey alone)
    uint8 tier2Threshold;    // e.g., 5 (passkey + BLS)
    uint8 tier3Threshold;    // e.g., 6 (passkey + BLS + guardian)
}
```

### Examples

**Default Personal (total possible = 10)**:
- Small tx (≤0.1 ETH): threshold 3 → passkey alone (weight 3) ✓
- Medium tx (≤1 ETH): threshold 5 → passkey (3) + BLS (2) = 5 ✓
- Large tx (>1 ETH): threshold 6 → passkey (3) + BLS (2) + guardian (1) = 6 ✓

**High Security (total possible = 10)**:
- Small tx: threshold 5 → passkey (3) + ECDSA (2) = 5 ✓
- Medium tx: threshold 7 → passkey (3) + ECDSA (2) + BLS (2) = 7 ✓
- Large tx: threshold 8 → passkey (3) + ECDSA (2) + BLS (2) + guardian (1) = 8 ✓

**Community DAO (total possible = 10)**:
- Any tx: threshold 4 → needs at least 2 different sources
- E.g., passkey (3) + any guardian (1) = 4 ✓
- Or: BLS (2) + 2 guardians (2) = 4 ✓

## Implementation Design

### New Signature Format (algId 0x06)

```
[0x06][sourceBitmap(1)][P256 sig?][ECDSA sig?][BLS payload?][guardian sigs?]
```

`sourceBitmap` indicates which signatures are present:
- bit 0: P256 passkey (64 bytes: r, s)
- bit 1: ECDSA (65 bytes: r, s, v)
- bit 2: BLS aggregate (variable)
- bit 3: Guardian 0 ECDSA (65 bytes)
- bit 4: Guardian 1 ECDSA (65 bytes)
- bit 5: Guardian 2 ECDSA (65 bytes)

### Validation Function (Prototype)

```solidity
function _validateWeightedSignature(
    bytes32 userOpHash,
    bytes calldata sigData
) internal view returns (uint256) {
    if (sigData.length < 1) return 1;

    uint8 sourceBitmap = uint8(sigData[0]);
    uint256 totalWeight = 0;
    uint256 offset = 1;

    // P256 passkey
    if (sourceBitmap & 0x01 != 0) {
        if (offset + 64 > sigData.length) return 1;
        if (_validateP256(userOpHash, sigData[offset:offset + 64]) != 0) return 1;
        totalWeight += weightConfig.passkeyWeight;
        offset += 64;
    }

    // ECDSA
    if (sourceBitmap & 0x02 != 0) {
        if (offset + 65 > sigData.length) return 1;
        if (_validateECDSA(userOpHash, sigData[offset:offset + 65]) != 0) return 1;
        totalWeight += weightConfig.ecdsaWeight;
        offset += 65;
    }

    // BLS aggregate (variable length)
    if (sourceBitmap & 0x04 != 0) {
        // Parse BLS payload length and validate
        // ...
        totalWeight += weightConfig.blsWeight;
    }

    // Guardian signatures
    for (uint8 i = 0; i < 3; i++) {
        if (sourceBitmap & (0x08 << i) != 0) {
            if (offset + 65 > sigData.length) return 1;
            bytes32 hash = userOpHash.toEthSignedMessageHash();
            address recovered = hash.recover(sigData[offset:offset + 65]);
            if (recovered != guardians[i]) return 1;
            totalWeight += weightConfig.guardianWeight[i];
            offset += 65;
        }
    }

    // (tier determined in _enforceGuard based on value)

    return totalWeight >= requiredWeight ? 0 : 1;
}
```

## Gas Considerations

- Each additional signature verification adds ~3000-6000 gas (ECDSA), ~3000 (P256 precompile)
- BLS verification is the most expensive (~50000+ gas)
- Weight calculation itself is negligible (~200 gas)
- Total overhead vs fixed tiers: ~500 gas for bitmap parsing + weight accumulation

## Migration Path

1. **Phase 1** (M4.1, done): Fixed cumulative tiers (algId 0x04, 0x05)
2. **Phase 2** (M5): Add weight-based validation (algId 0x06)
3. **Phase 3** (M6): Frontend config UI for weight customization

The fixed tier model (0x04/0x05) remains as efficient shortcuts for the most common patterns. Weight-based (0x06) is for advanced users who need custom security profiles.

## Open Questions

1. **Should passkey always be required?** Currently yes (weight 3 is always needed for threshold). But some community DAOs might want guardian-only governance.
2. **Weight update governance**: Who can change weights? Owner only? Or require guardian approval for security-critical changes?
3. **Maximum weight per source**: Should we cap individual weights to prevent a single compromised source from reaching threshold alone?
4. **On-chain vs off-chain threshold lookup**: Store threshold per tier in contract, or let client compute and validate?

## Conclusion

Weight-based signatures provide maximum flexibility while maintaining the cumulative security principle. The bitmap approach keeps signature format compact and gas-efficient. Implementation should be deferred to M5 after the fixed tier model is battle-tested in production.
