# 隐私优先的多重验证 ERC-4337 智能钱包：产品与技术架构设计思路
背景概述：我希望设计一个智能合约账户体系，包括以下features：
- 适配以太坊的发展路线，基于ERC-4337,未来合并到Native AA。
- 服务于普通个体和社区的使用场景，包括为个体的Agent提供子账户和授权进行打赏、支付、社交等操作。
- 可选的隐私优先，和未来的抗量子适配。
- 提供多签、社交恢复，多层的安全机制：passkey，混合多签、合约兜底等等功能。
- 如通自己的钱包，如果旧了，把钱拿出来，换一个新钱包即可，随时可以搬家。
- 另外对于EOA，提供了基于EIP-7702的桥接机制，可以安全简单便捷的享受部分AA的便利功能。
- 另外，我希望把这个项目，拆分为三个论文：AA的多签核心，EOA的桥接机制，隐私优先的智能钱包，在DSR仓库对应了Paper4,5,6。
    

## 1. 产品概述与理念设定

本产品的核心愿景是构建一个 **符合 ERC-4337 账户抽象标准、隐私优先、不可升级且高度安全的个人多签智能钱包**。
它将彻底摆脱传统钱包的单点故障与隐私泄露风险，融合最前沿的密码学技术（如 ZK-SNARKs、WebAuthn），为终端用户提供“随着资金暴露风险增加而动态升级安全策略”的渐进式安保体验。

基于提供的 `stackup simple-team-account` 与 `alchemy light-account` 代码库，以及 `zerodev kernel` (行业领先的 ERC-7579 模块化智能账户框架)，结合对 Railgun、Kohaku（锦鲤 SDK） 等技术文档的深度分析，我们确立以下产品的核心特质：
1. **彻底的不可升级性（Non-upgradability）**：在合约层摒弃 UUPS 等代理升级模式。任何业务迭代都通过部署新版本合约和引导用户资产跨合约迁移实现，彻底根绝由于管理员或部署者被黑导致的安全漏洞。
2. **渐进式分级安全验证**：基于用户的转账与用钱额度自动调整验证强度的多签机制（如小于 $100 无缝通行，超出则触发层层联防）。
3. **不可逾越的底层红线（Immutable Global Guards）**：在合约底层通过硬编码逻辑注入核心限制（例如每日不可变更的 $500 转账风控额度），即使获得最高权限的签名也无法突破该物理约束。
4. **全栈隐私支持**：底层原生地拥抱 Railgun 等隐私中间件架构，能通过中继与 ZK-SNARKs 屏蔽所有者的身份、余额和操作路径。

## 2. 需求分析与模块化拆解

根据您的需求，产品的结构需极为清晰并高度模块化，因此我们将钱包组件解耦为核心的主状态机与多个可插拔的外围验证和执行模块：

### 2.1 签名与验证模块（Validators）
验证模块**必须被设计为完全解耦且可动态热插拔的独立合约/插件**，用作判断“某笔交易是否合法”的标准器。由于项目未来需引入锦鲤 SDK 并支持抗量子密码学，验证层的原生模块化是重中之重：
- **P-256 (WebAuthn / Passkey) Validator**：集成移动设备的原生硬件安全区（Secure Enclave），使用指纹或面容等生物特征在本地进行签名。这是中小额高频操作的主力验证流。
- **K1 (Secp256k1) 传统 Validator**：对以太坊传统算法的后向兼容支持，便于接入现有的以太坊生态签名器。
- **未来扩展：Post-Quantum (抗量子) Validator**：预留标准的 Validator 接口，以便未来平滑引入基于哈希（Hash-based）或格密码（Lattice-based）的抗量子签名算法进行验证。
- **未来扩展：DVT / 外部网关（Verifier） Validator**：类似 `simple-team-account` 中“Verifier”的设计，如果发生大额交易或敏感操作，不仅需要用户的设备，还需要一个去中心化的守护节点（Guard Nodes）进行交叉签名。

### 2.2 多层级安全风控路由
对于金额大小的验证机制（基于预设或动态设定的美金额度，假设挂钩预言机或在建立时刻锚定）：
- **Tier 1 (比如 < 100 USD)**：单因素生物识别（仅仅 WebAuthn Passkey 即可放行）。
- **Tier 2 (比如 100 - 1000 USD)**：双重因素验证，例如 `Owner的 Passkey + 第二台设备的 Passkey`，或者 `Owner Passkey + 一个特定的 EOA 签名`。
- **Tier 3 (比如 > 1000 USD)**：多重签名（3/3 或 2/3），需要 Owner 设备、备份设备，以及外挂的 DVT 服务器或硬件钱包等共同出具签名。这可能采用链下的阈值签名聚合（Threshold Signature）在链上体现为一个简单的组签名。

### 2.3 绝对底座：红线护卫器（Global Guards）
有别于常规的安全配置可被随意更改，该组件旨在提供“极端的最后一道防线”。
- **硬编码创建规则**：在部署创建当前版本钱包实例时，即写入 `DAILY_LIMIT_USD`（或者特定代币计价），这部分状态及验证逻辑放置在钱包根合约的最核心，任何 Validator 模块和任何 Execute 操作必须穿透该 Guard。一旦触及单日极限，任何级别的联名上书（多签）都将宣告 Revert。

### 2.4 原生隐私整合（Railgun / Kohaku SDK）
- **隐私存储（Shielded Pool）**：用户的大部分个人资本实际停留在 Railgun 的加密 UTXO 树中（即 Shield 状态），而非公开直接挂在钱包对应的 EOA/Contract 地址上。
- **执行代理（Relay Adapt & Executor）**：用户需要交易时，通过零知识证明向网络出具证明，驱动 Relay Adapt 合约将其在暗池中的有效资金释放进行即时兑换 / 转账，然后再度隐匿（Reshield）。
- **OAPD (One Account Per DApp)**：借助钱包内的特定代理模块生成对应的隔离子账户，避免大额主账户身份暴露。

## 3. 技术架构设计方案与底座选型（Smart Contract 层面）

在吸收了 `simple-team-account` (极简、角色权限控制)、`light-account` (高可用与稳定) 以及 `zerodev kernel` (高度模块化的 ERC-7579 标准) 的模式下，我们需要在**“模块化扩展性”**与**“代码极简、接口直接”**之间做出权衡。以下是三种潜在底座方向的对比分析，本项目的最终架构将通过讨论后敲定。

### 3.1 潜在核心底座对比

| 评估维度 | 纯手工极简构建 (参考 Simple-Team-Account) | 基于 Alchemy Light-Account (ERC-6900/4337) | 基于 ZeroDev Kernel (ERC-7579) |
| :--- | :--- | :--- | :--- |
| **代码极简性与模块数** | **最高**。仅包含我们实际需要的验证、递进逻辑和定死的风控限制（全局红线），几乎没有多金的接口暴露。 | **高**。Light-Account 是一个针对单一签名的轻量化 4337 抽象账户。非常干净，但其设计初衷是单 Owners 轮换，原生地不包含多级权限和复杂的执行层。 | **较低**。Kernel 包含了完整的 ERC-7579 安装、卸载模块钩子、回退处理等复杂逻辑，代码量大且对审计要求高。 |
| **功能贴合度 (多层验证 / 权限路由)** | **完美贴合**。硬编码解析不同交易金额触发的多签策略和不同的 Validator，没有任何历史包袱。 | **较弱**。Light-Account 验证逻辑写死在了 `validateUserOp` 和 `isValidSignature` 里面。需要我们“魔改”它的基础验证流程来注入“按金额分级”的多签。 | **原生支持但冗杂**。支持海量复杂的验证模式组合。 |
| **安全性与不可升级机制** | 逻辑直白，天然适合做**彻底的不可升级设计**。代码接触面最小化，可以把底层防御护城墙筑得最厚。 | Alchemy 代码经过了严苛的审计（如 UUPS 代理和存储控制）。若取消其升级，我们可以得到一个非常安全的基座，但要修改其 ECDSA 验签部分引入 WebAuthn。 | 可以通过锁死权限实现不可升级，但底层 ERC-7579 庞大的状态机机制增加了被攻击的面数风险。 |
| **未来扩展性** | **较弱**。除了预留的 DVT 或限额风控插槽外，无法热插拔新功能。升级只能靠发新版迁移资产。 | **中等**。它本质是一个不可扩展的单体合约（没有 Plugin / Module 概念）。只能整体替换。 | **极强**。任何人都可以为其开发适配标准的新插件。 |

**团队核心指导思想探讨与敲定（极简 + 强制验证模块化）：**
经过评估，既要满足“代码极简，接口直接”，又必须满足**“面向锦鲤SDK和由于引入抗量子的格密码（Lattice-based）/哈希（Hash-based）签名而必须做到的签名与验证的绝对模块化”**。

我们最终倾向的折中方案：
**手工打造一个裁剪版的 ERC-7579 (或类似架构) 作为核心状态机**。
1. 主机（Core Account）极度精简，**仅提供 `validateUserOp` 和 `execute` 的路由能力**。
2. 剥离所有的签名验证逻辑，**强制要求所有签名验证必须指向独立的 Validator 插件**。
3. 把安装控制权和多级路由控制权做硬编码限制（例如：通过全局护卫网关，写死当金额 > 1000U 时，强制路由至由 `WebAuthn Validator` + `PQ Validator` 组成的多签聚合器），这样就不需要引入 Kernel 里巨大且危险的动态插件生命周期管理（安装/卸载钩子）。

### 3.2 深度设计细节

**1. 账户隔离与创建过程 (Factory & Non-upgradable)**
- 自行编写并优化基于 ERC-4337 `BaseAccount` 的极简工厂合约，使用 CREATE2 部署。去除所有的代理模式，部署的就是最终逻辑合约。
- 初始化参数直接包含核心 Validator 地址和不可变的全局限额（如每日 `$500`）。这样省去了复杂的安装卸载钩子逻辑。

**2. 阶梯验证（Tiered Multisig）的技术实现**
- 在验证 `UserOperation.signature` 时，解码该字段获取意图执行的等级以及连带的所有签名集。
- 根据 `UserOperation.callData` 评估本次操作可能涉及的价值和资产。
- 逻辑：如果转出资产 <= 100 USDC，则仅校验 Signature Part 1 是否为 Owner WebAuthn。如果 > 100 USDC，强制要求包含 Signature Part 2 (Verifier / Second Owner)，如果遇到非常大额，强制要求 Signature Part 3 (DVT多签验证聚合签名)。

**3. 对接隐私层：Railgun / Kohaku SDK**
- 账户架构中，需要支持 `DelegateCall` 或者专用执行出口到 `Relay Adapt` 协议模块。
- 即：用户的核心隐私资金并不暴露在智能账户的可见余额上，而是保存在 Railgun 池内（拥有 ZK 资金证明）。
- 当智能账户向需要公开调用的 DApp 发送资金时（如通过子账户交互），调用路径为：`EntryPoint -> Core Account -> RailgunExecutor -> Railgun Shielded Pool (进行 Unshield 操作并原子级执行给子账户)`。
- 通过在移动端接入 Kohaku SDK 进行 ZK 证明的生成，然后推入 UserOperation；钱包智能合约仅充当状态机器，管理那些只有经过授权并解封的原子资金。

## 4. 后续落地与研发前瞻

1. **版本更迭与状态转移**：由于严格定义其为**不可升级**，当我们需要引入新的隐私池标准或替换存在漏洞的组件时，团队需发布 v2 版本的 Factory 和组件。用户端在 UI 界面一键触发 `MigrateToV2` 的原子批处理 UserOp：部署 v2 钱包实例 -> 解绑转移所有的受控资产 (Unshield / Transfer) -> 将控制权过渡给 v2 -> 销毁原账户多余逻辑 (选做)。这种方式极为干净、清晰且彻底消除了管理员后门风险。
2. **WebAuthn 的 Gas 优化**：当前的纯 Solidity P-256 签算非常耗费 Gas (大约 72K-300K Gas 取决于实现)，项目组在部署时需要结合所采用链是否支持 EIP-7212 (P256预编译) 或采取聚合优化打包机制（通过锦鲤/Paymaster全额补贴）。
3. **架构审计与隔离**：必须将“额度硬控制”（Global Guards）与“权限路由”（Validators）在代码层做最严格的边界切割，不给外挂挂件或者甚至所有者本人的单一疏忽留有能绕过限额防御代码的机会。
